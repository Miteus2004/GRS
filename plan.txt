Overall Strategy
Build the engine inside-out: first the data model and code skeleton (pure Python, no infrastructure needed), then the templating and config generation (still offline), then wire it all up in the lab. The SDN layer follows the same pattern — write the Ryu logic outside, integrate with OVS inside.

Phase 1 — Core Engine ✅
🏠 Outside the lab
✅Step 1 — YAML schema design
This is the most important design decision you'll make. Before writing a single line of engine code, define exactly what the YAML looks like. Sketch out the four sections (organization, networks, routing, services) with real example values. Lock this down first — everything else derives from it.
Concrete draft: see intent.example.yaml.
✅Step 2 — Python engine skeleton
Create the main entry point and the module structure. A clean layout would be something like engine/parser.py (reads and validates the YAML), engine/renderer.py (Jinja2 template rendering), engine/provisioner.py (Netmiko calls), and engine/monitor.py (Nagios polling). You can write and unit-test the parser and renderer entirely offline.
Initial scaffold: see engine/ and pyproject.toml.
✅Step 3 — Jinja2 templates
Write the templates for ospfd.conf, zebra.conf, named.conf, BIND9 zone files, and the Nginx upstream block. These are just text files with variables — you can render them locally and inspect the output without any network device. Write a small test script that loads your example YAML and renders each template, then manually verify the output looks correct.
✅Step 4 — Docker Compose topology
Design your docker-compose.yml with all the containers: routers (Quagga), DNS server, Nginx, Nagios, and later the Ryu controller. You can write and validate this file offline — docker-compose config will parse and validate it even if you don't start anything. If you have Docker locally, you can even spin up the containers and verify they start, even without the actual routing config inside them.
✅Step 5 — BIND9 and Nginx auto-generation logic
The Python functions that take the YAML and produce zone files and upstream blocks are pure logic — no lab needed. Write these, test them with your example YAML, and make sure the output files are syntactically valid (you can run named-checkzone and nginx -t locally if you have them installed, or just inspect manually).

🔬 Inside the lab
Step 6 — Netmiko SSH provisioning
With the rendered config files ready, now you connect. Netmiko SSHs into each router container and sends the configuration commands. Start with a single router and a single config file. Verify the config was applied correctly before moving to the full topology.
Step 7 — OSPF and BGP validation
Bring up the full Quagga topology. Use vtysh to inspect routing tables and verify adjacencies formed correctly. Test reachability between subnets. This is iterative — you'll likely tweak your Jinja2 templates here based on what Quagga actually needs.
Step 8 — BIND9 and Nginx live test
Deploy the generated zone files into the BIND9 container and test DNS resolution. Deploy the Nginx upstream config and verify load balancing across the four replicas with curl and checking access logs.
Step 9 — Nagios closed loop
Configure Nagios with the auto-generated host definitions. Verify it detects hosts correctly. Then write the feedback loop in the engine: poll Nagios → parse status → decide whether action is needed. You can simulate a failure by stopping a container and watching the loop react.

Phase 2 — SDN Layer
🏠 Outside the lab
Step 10 — Ryu controller application
The Ryu app is pure Python. Write the application that handles OpenFlow events and exposes a REST API for the IBN engine to push flow rules. You can test this logic with Ryu's built-in testing tools or even just unit-test the REST handler functions without a real switch.
Step 11 — Congestion detection logic
The piece of the engine that polls Nagios metrics, compares against thresholds, and decides to push new flow rules is pure logic. Write and test this offline with mocked Nagios responses.
Step 12 — Flow rule definitions
Define your primary and backup path flow rules in Python data structures. Map them to the topology you designed in the Docker Compose file. This is conceptual work you can do anywhere.

🔬 Inside the lab
Step 13 — OVS integration
Add Open vSwitch to the router containers in your Compose file and connect it to the Ryu controller. Verify that Ryu sees the switch connect and installs a basic flow rule. This is where things get fiddly — plan for a full session just on this step.
Step 14 — Primary and backup paths
Install the flow rules for the primary path and verify traffic follows it with traceroute. Then install the backup rules and verify the switch.
Step 15 — Congestion demo
Use tc netem delay to inject artificial latency on a link. Watch the engine's monitoring loop detect it via Nagios, call the Ryu REST API, and push the new flow rules. Verify with traceroute that traffic moved to the backup path. Time it — the target is under one second.

Suggested File Structure
ibn-engine/
├── intent.yaml              ← your network declaration
├── docker-compose.yml       ← full topology
├── engine/
│   ├── main.py              ← entry point, reconciliation loop
│   ├── parser.py            ← YAML loading + validation
│   ├── renderer.py          ← Jinja2 template rendering
│   ├── provisioner.py       ← Netmiko SSH calls
│   ├── monitor.py           ← Nagios polling
│   └── sdn.py               ← Ryu REST API calls
├── templates/
│   ├── ospfd.conf.j2
│   ├── zebra.conf.j2
│   ├── named.conf.j2
│   ├── zone.j2
│   └── nginx_upstream.j2
├── ryu_app/
│   └── ibn_controller.py    ← Ryu OpenFlow application
└── tests/
    └── test_renderer.py     ← offline unit tests
